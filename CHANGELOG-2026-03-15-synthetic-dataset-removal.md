# Changelog: Synthetic Dataset Removal & Notebook Cleanup

**Date:** 2026-03-15
**Scope:** Training data pipeline, notebook inference config
**Impact:** All future model retraining will produce significantly higher quality playbook output

---

## Why This Change Was Made

### The Problem

The fine-tuned Llama 3.1 8B model was producing poor incident response playbooks:

- Outputs described the **attacker's kill chain** instead of the **defender's response** (e.g., listing Metasploit and Mimikatz as "tools," phases named "Initial Compromise" and "Data Exfiltration")
- Actions were **vague and non-actionable** ("Remove malicious files," "Conduct incident response," "Restore systems")
- **Generic placeholder tool names** instead of real products ("Incident Response Software," "Malware Removal Tool," "System Restore Tool")
- Responses were **truncated mid-sentence** due to low token budget
- **Missing NIST IR phases** — most outputs lacked Recovery or Lessons Learned

### Root Cause: Training Data Contamination

The model was trained on **692 combined examples** from two datasets:

| Dataset | Rows | Quality |
|---------|------|---------|
| `ir_playbooks_alpaca.jsonl` (primary) | 173 | 100% HIGH |
| `ir_playbooks_synthetic_alpaca.jsonl` (synthetic) | 519 | 99.2% LOW |

The synthetic dataset — which made up **75% of training data** — was generated by Llama 3.2 3B and had systemic quality failures. The model learned the majority pattern (the bad data) rather than the minority pattern (the good data).

### The Audit

A comprehensive programmatic audit was run across both datasets. Key findings for the **synthetic dataset**:

**Generic Tools: 66% of all tool references were placeholders**
- 2,784 generic tool names out of 4,221 total references
- 120 entries (23%) contained **only** generic tools with zero real product names
- Top offenders: "Incident Response Software" (163x), "System Restore" (86x), "Antivirus Software" (77x)

**Missing NIST Phases:**
- 0% of entries had all 5 standard phases (Identification, Containment, Eradication, Recovery, Lessons Learned)
- 62.6% missing Recovery
- 56.8% missing Eradication
- 37.8% missing Lessons Learned

**Phase Name Chaos: 130+ unique phase names** including nonsensical ones:
- "Initial Response" (425x) used instead of "Identification"
- "Incident Response" (112x) — meaningless as a phase name
- "Eradiation" (88x) — misspelled "Eradication"
- "Ransomware Negotiation," "Evasion Techniques," "Content Delivery," "Ecosystem Response"

**Attacker-Perspective Entries: 28 entries (5.4%)**
- Phases named "Data Exfiltration Containment/Prevention/Response" frame the attack chain as the response structure

**Vague Actions: 41% of entries**
- "Alert and Contain" (105x), "Notify Management" (14x), "Restore Systems" (16x)

**0-Minute Response Times: 20% of entries**

**Near-Duplicates: 55% of entries** had the same incident type + target asset as another entry (e.g., "data breach / customer database" appeared ~40 times with different but uniformly low-quality outputs)

In contrast, the **primary dataset** scored perfectly on every metric: 100% real tools, 100% all 5 NIST phases, 0% vague actions, 0% attacker perspective, 0% truncation.

### The Decision

Rather than attempt to fix 515 low-quality entries, the synthetic dataset was dropped entirely. The 173 primary entries are structurally correct and consistent. Retraining on primary-only will produce dramatically better output even with fewer examples, because every example teaches the correct pattern.

## Known Remaining Issues

1. **55 exact duplicates in primary dataset** — 173 rows contain only ~118 unique examples. Deduplication has not been performed yet. This causes the model to over-memorize certain incident types (e.g., ransomware, phishing) at the expense of less-represented types.

2. **Small dataset size** — 173 rows (118 unique) is small for SFT. If more volume is needed, synthetic data should be regenerated using a stronger model (Claude, GPT-4) with explicit constraints: must use the 5 NIST phase names exactly, must reference real named tools, must include specific actionable steps, must be from the defender's perspective.

3. **RAG knowledge base** — The chatbot's `RAG-Chatbot/data/` directory currently contains only a single PDF (`K-Maps Lecture Notes page.pdf`) which is not IR-related. Adding IR-specific documents (NIST SP 800-61, MITRE ATT&CK docs, org-specific runbooks) would significantly improve retrieval-augmented responses.

# Changelog: Local RTX 5070 Migration Attempt & Colab Revert

**Date:** 2026-03-16
**Scope:** Training notebook runtime environment
**Impact:** Notebook reverted to Colab after Blackwell GPU incompatibility with Unsloth's attention dispatch

---

## What Happened

### Goal

Migrate the training notebook from Google Colab to a local RTX 5070 GPU (12 GB VRAM, Blackwell architecture, compute capability 12.0) to avoid Colab session limits and cost.

### Changes Made (Then Reverted)

All of the following changes were applied to `IRP_agent_trainint.ipynb` and subsequently **reverted** when training failed:

| Cell | Local Change | Reason |
|------|-------------|--------|
| #0 (markdown) | Replaced Colab instructions with conda env setup for `unsloth_env` (Python 3.12, PyTorch cu128) | Local environment needs manual setup |
| #2 (env check) | Replaced Colab detection with `torch.cuda.get_device_properties()` GPU info | No Colab runtime to detect |
| #3 (installs) | Replaced `%%capture`/`%pip` with `subprocess.check_call` + xformers uninstall | Jupyter magics don't work outside Colab; xformers incompatible with Blackwell |
| #7 (LoRA config) | `use_gradient_checkpointing = "unsloth"` | More VRAM-efficient than `True` (kept in final revert — works on Colab too) |
| #10 (data prep) | Absolute Windows path `r:\Dropbox\_Code\Projects\irp\datasets\...` replacing Google Drive mount | Local filesystem access |
| #14 (trainer) | `batch_size` 2→1, `gradient_accumulation` 4→8 | Fit within 12 GB VRAM while keeping effective batch = 8 |
| #21 (save) | Absolute path `r:\...\irp_exports\ir_assistant_lora` | Save directly to project dir |
| #23 (verify save) | Absolute path check | Match cell #21 |
| #24 (markdown) | "Verify Exported Adapters" replacing "Export to Google Drive" | No Drive on local |
| #25 (export) | Local file existence + size verification replacing Drive mount/copy | No Drive on local |
| #27 (test) | Local adapter path | Load from local save |

### The Failure: xformers on Blackwell

Training crashed at `trainer.train()` with:

```
NotImplementedError: No operator found for `memory_efficient_attention_forward`
  `fa3F@0.0.0` is not supported because:
      requires device with capability <= (9, 0) but your GPU has capability (12, 0) (too new)
  `cutlassF-pt` is not supported because:
      requires device with capability <= (9, 0) but your GPU has capability (12, 0) (too new)
```

**Root cause:** Unsloth installs xformers as a dependency and auto-routes attention through it. All xformers attention kernels (FA2, FA3, CUTLASS) cap out at compute capability 9.0. The RTX 5070 (Blackwell, sm_120, compute capability 12.0) has no compatible xformers kernels.

**Attempted fix:** Uninstall xformers (`pip uninstall xformers -y`) so Unsloth falls back to SDPA (PyTorch's native scaled dot-product attention). This was added to Cell #3 and to the setup instructions in Cell #0.

**Why it didn't work:** Despite uninstalling xformers via pip, the error persisted. Likely causes:
1. The Jupyter kernel had already imported xformers into memory during model load (Cell #5), and uninstalling the package mid-session doesn't remove in-memory modules
2. Unsloth's compiled cache (`unsloth_compiled_cache/`) may have baked xformers dispatch paths that survived the uninstall
3. The cache directory was locked by the running kernel and couldn't be deleted

A full kernel restart + cache deletion + re-run from Cell #2 should theoretically have resolved this, but the user opted to revert to Colab instead.

### The Revert

All cells were restored to Colab-compatible versions:
- `%%capture`/`%pip` install magics
- Google Drive mount for dataset access (`My Drive/irp/datasets/ir_playbooks_alpaca.jsonl`)
- Colab-appropriate batch size (2) and gradient accumulation (4)
- Relative save paths (`ir_assistant_lora/`)
- Google Drive export (folder + zip to `My Drive/irp_exports/`)
- Test cell loads from `/content/drive/MyDrive/irp_exports/ir_assistant_lora`

**One change kept from the local attempt:** `use_gradient_checkpointing = "unsloth"` in Cell #7. This is an Unsloth-specific optimization that uses less VRAM than `True` and works on all platforms.

### Training Completed on Colab

After reverting, training completed successfully on Colab.

---

## Lessons Learned

1. **Blackwell (RTX 50-series) is too new for xformers** — As of March 2026, xformers 0.0.35 has no kernels for compute capability > 9.0. Unsloth silently depends on xformers for its attention dispatch.

2. **Uninstalling a package mid-Jupyter-session is unreliable** — Python caches imported modules. If xformers was imported before it was uninstalled, the in-memory module persists. The fix requires: uninstall → restart kernel → delete compiled caches → re-run.

3. **Unsloth's compiled cache can persist stale dispatch paths** — The `unsloth_compiled_cache/` directory stores JIT-compiled trainer code. If generated with xformers present, it may continue routing to xformers even after uninstall.

4. **For future local RTX 5070 attempts:** Ensure xformers is uninstalled *before first import of unsloth*, delete `unsloth_compiled_cache/`, and verify the Unsloth banner shows `FA [Xformers = None]` before proceeding to training.

# Changelog: RAG-Chatbot Post-Retrain Fixes

**Date:** 2026-03-16
**Scope:** `RAG-Chatbot/app.py`, `.env.example`, `.env`
**Impact:** Chatbot inference now uses the correct Alpaca prompt format matching the fine-tuned model's training data

---

## What Changed

### 1. Critical Fix: Prompt Template Mismatch (`app.py` — `build_prompt()`)

**The Problem:**

The fine-tuned model was trained exclusively on the Alpaca prompt template:
```
Below is an instruction that describes a task, paired with an input that provides further context.
Write a response that appropriately completes the request.

### Instruction:
{instruction}

### Input:
{input}

### Response:
```

But the chatbot's `build_prompt()` function was sending prompts in a completely different format:
```
You are an incident response assistant. Use the retrieved context to answer accurately.
If context is insufficient, say what is missing and provide a best-effort response.

Context:
{context_text}

Question:
{user_question}

Answer:
```

The model had **never seen this format during training**. This format mismatch would cause significantly degraded output quality — the model wouldn't recognize the prompt structure and would fail to activate the fine-tuned behavior.

**The Fix:**

`build_prompt()` now wraps user queries in the Alpaca template the model was trained on:
```python
def build_prompt(user_question: str, context_text: str) -> str:
    instruction = (
        "Generate an incident response playbook for the following security incident. "
        "Use the retrieved context to inform your response where relevant."
    )
    input_text = user_question
    if context_text and context_text != "No relevant context retrieved.":
        input_text += f"\n\nRetrieved context:\n{context_text}"

    return (
        "Below is an instruction that describes a task, paired with an input "
        "that provides further context. Write a response that appropriately "
        "completes the request.\n\n"
        f"### Instruction:\n{instruction}\n\n"
        f"### Input:\n{input_text}\n\n"
        "### Response:\n"
    )
```

The RAG-retrieved context is appended to the `### Input:` section so the model sees its trained format while still benefiting from retrieved documents.

### 2. MAX_NEW_TOKENS Updated (`.env.example`)

- **Before:** `MAX_NEW_TOKENS=384`
- **After:** `MAX_NEW_TOKENS=512`
- **Why:** The training notebook's inference cells were updated to 512 tokens to prevent truncation of multi-phase playbooks. The chatbot default should match.

### 3. `.env` File Created

- `.env` was missing entirely (only `.env.example` existed). The app would crash on startup with `ValueError: LORA_ADAPTER_PATH not found`.
- Created `.env` with:
  - `LORA_ADAPTER_PATH` pointing to `irp_exports/ir_assistant_lora`
  - `MAX_NEW_TOKENS=512`
  - All other defaults matching `.env.example`

---

## Why This Matters

Prompt format is one of the most impactful factors in fine-tuned model quality. A model trained on format A and prompted with format B will:
- Fail to activate the fine-tuned LoRA weights properly
- Fall back to base model behavior (generic, untrained responses)
- Produce outputs that don't follow the learned NIST phase structure

With this fix, the chatbot now sends prompts in the exact Alpaca format the model learned during SFT, so the fine-tuning investment actually carries through to the user-facing application.
